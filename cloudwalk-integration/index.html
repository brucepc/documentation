Introduction

API Endpoint
https://txg.sumup.com/v1/cloudwalk-payments

The purpose of this document is to describe how to integrate card readers with SumUp payments backend.

This document is a work-in-progress, meaning that messages, endpoints, specifications and any other information contained in this document may change without any notice. Be advised.

Transport and security ¶

The communication protocol used is HTTP over TLS and the HTTP verb used is always POST. The content of the exchange messages must be sent in the body of the request and the Content-Type header must be application/xml and the charset is utf-8.

Every HTTP request must contain a header called Serial-Number containing the serial number of the device.

Messages based in the 2.1 version of the ISO-20022 standard will be exchanged between the reader and the SumUp payments backend in order to process transactions.

Sensitive card data (such as the PAN and PIN block) must be encrypted using DUKPT 3DES. The PIN block must have it’s own encryption key which is not to be used to encrypt the PAN or any other information. In other words, two keys must be used by the reader, one to encrypt the PAN and other sensitive data (if any) and another to encrypt the PIN block.

Data Types ¶

As part of the messages, the following data types will be used:

  • Text: Regular text;

  • Boolean: Boolean values (true/false);

  • Amount: Used for numeric values containing decimals;

  • DateTime: Date and time following the ISO-8601 standard (yyyy-MM-DDTHH:mm:ss-03:00). The date time field must contain the timezone information;

  • CodeSet: Used for pre-defined values in the descriptions;

  • Container: Groups other elements;

Messages ¶

Messages exchanged between readers/client systems are segregated in two types - ISO-20022 standard messages and SumUp specific messages.

SumUp specific messages:

  • SumUpActivationRequest - Request used to activate a device

  • SumUpActivationResponse - Response for activation

  • SumUpReceiptRequest - Request used to send a receipt for a transaction (currently only SMS is supported)

  • SumUpReceiptResponse - Response for receipt

ISO-20022 standard messages:

  • AcceptorAuthorisationRequest - Request used for authorising transactions

  • AcceptorAuthorisationResponse - Response for authorisation requests

  • AcceptorCancellationRequest - Request used for refund or reverse a transaction

  • AcceptorCancellationResponse - Response for refund and reversals

  • StatusReport - Used to manage the device tables

  • ManagementPlanReplacement - Used to manage the device tables

  • AcceptorConfigurationUpdate - Used to manage the device tables

Even though SumUp uses the ISO-20022 standard, there are some minor modifications to the standard specification, such as supporting PAN encryption and so on.

Activate a device
POST/

This endpoint should be called to associate a device with a merchant in the SumUp backend.

Devices that are not yet activated won’t be able to transact, meaning that authorization messages will be declined by SumUp backend.

SumUpActivationRequest message fields:

Attribute (TagName) Type Description
* Header(Hdr) [1…1] Container Header of the message
** MessageFunction(MsgFctn) [1…1] CodeSet Identifies the type of the message

NOTE: Fixed value - “SARQ” = SumUp Activation Request
* ActivationCode (ActivationCode) [1…1] Text Activation code (entered by merchant)
* DeviceSerialNumber (DeviceSerialNumber) [1…1] Text Serial number of the device being activated

SumUpActivationResponse message fields:

Attribute (TagName) Type Description
* Header(Hdr) [1…1] Container Header of the message
** MessageFunction(MsgFctn) [1…1] CodeSet Identifies the type of the message

NOTE: Fixed value - “SARP” = SumUp Activation Response
* Success (Success) [1…1] Bool Indicates if the activation was successful
* Code (Code) [1…1] CodeSet SumUp’s return code

“0” = success
“-1” = invalid activation code
“-2” = code already used
“-3” = device already activated

Send a receipt
POST/

The receipt message can be sent in two moments:

  • After a successful transaction;

  • When the merchant wants to resend a receipt;

It receives the transaction ID generated by the client system/device and the cardholder’s information.

SumUpReceiptRequest message fields:

Attribute (TagName) Type Description
* Header(Hdr) [1…1] Container Header of the message
** MessageFunction(MsgFctn) [1…1] CodeSet Identifies the type of the message

NOTE: Fixed value - “SRRQ” = SumUp Receipt Request
* InitiatorTransactionIdentification (InitrTxId) [1…1] Text Transaction unique ID generated by the client

NOTE: Must be equal to InitrTxId used in the authorisation request
* Mobile Phone Number (MobilePhoneNumber) [0…1] Text Cardholder’s phone number

NOTE: Format is 5511999998888
* E-mail (Email) [0…1] Text Cardholder’s e-mail

SumUpReceiptResponse message fields:

Attribute (TagName) Type Description
* Header(Hdr) [1…1] Container Header of the message
** MessageFunction(MsgFctn) [1…1] CodeSet Identifies the type of the message

NOTE: Fixed value - “SRRP” = SumUp Receipt Response
* Success (Success) [1…1] Bool Indicates if the receipt was sent
* Message (Message) [0…1] CodeSet Message to be displayed by the device

Download tables/parameters
POST/

To transact, the device/system that is communicating with SumUp must first update it’s parameters.

The flow is quite simple, the terminal communicates with SumUp informing it’s version of the parameters through a StatusReport and SumUp replies with a plan to update the configuration (ManagementPlanReplacement), which the device/system should follow.

The following parameters should be loaded into the system/device:

  • AID tables

  • CAPK tables

  • BIN ranges tables

  • Card brands tables

All of the tables are properly explained in more details bellow. The complete flow should be like:

  • Client device/system sends a StatusReport message to inform it’s version of the parameters;

  • SumUp replies with a ManagementPlanReplacement containing the list of actions to be performed, including the action to download the new parameters;

  • Client device/system sends another StatusReport requesting the new version of the parameters;

  • SumUp replies with an AcceptorConfigurationUpdate containing the new version of the parameters;
